Where the council agrees

A full big-bang migration now is off the table — all five agree two devs with no ops team cannot safely run a simultaneous live auth + data cutover. Doing nothing forever is also rejected (the debt compounds). Auth and data are different risk classes and must not move on the same timeline. Execution, if any, should be incremental and reversible — never a cutover weekend.

The recommendation

Don't migrate now. Don't commit to a vague “later.” Stop treating this as a two-option choice. The strongest reasoning — ranked highest by every peer reviewer — is the First-Principles reframe: solve the two real pains without touching Auth or the write path. Stand up a Postgres companion (even boring Cloud SQL / Firebase Data Connect) fed by CDC/Cloud Functions for relational queries in ~2 weeks with a fully reversible blast radius, and treat “predictable cost” as a measurement problem first. Escalate to a full Auth+data migration only if the companion hits a real wall, a third engineer/ops capacity exists, and the auth-hash path has been tested end-to-end against a prod copy.

Full responses

The Contrarian Don't — negative risk-adjusted payoff unless the pain is already acute. sonnet›

“Migrate later once bigger” is the trap — bigger is worse, not better. Every input to migration difficulty gets strictly harder with scale: more write traffic means a longer unsafe cutover window; more data means longer backfill; more features on Firestore's document model means more denormalized data to reshape — the actual hard part. “Better-resourced later” is unearned: no funding event, no headcount plan; “later” is a wish. Companies that defer infra migrations under growth do them under pain at 3× cost after an incident.

Also unquestioned: will Postgres/RLS actually solve the problem? RLS with two devs writing their own policies is a minefield — silent data leaks from a missed policy are worse than a bad NoSQL query because they're invisible until a breach. Edge Functions aren't drop-in for Cloud Functions. This isn't “swap the database,” it's “rewrite the backend,” sized like a data migration.

Fatal flaw: two devs, no ops, migrating live auth+data for money and convenience — not because Firebase is failing — is a negative risk-adjusted payoff unless the current pain is already acute. If it isn't, this is optimizing a working system into a broken one.

The First-Principles Thinker Wrong question — a reversible Postgres companion gives 90% of the want at 5% of the risk. sonnet›

What are we actually solving for? Not “Firebase vs Supabase.” The stated pain is “relational queries and predictable cost” — two different problems wearing one trenchcoat, and migrating the entire backend is wildly disproportionate to both.

Relational queries: you don't need to leave Firebase for Postgres. Run Postgres as a read-side companion, sync the subset that needs joins via Cloud Functions or CDC, and query that — relational power without touching Auth or the write path. A two-week project, not two quarters.

Predictable cost: almost never a migration problem — it's a query-pattern problem (N+1 reads, unbounded listeners, whole-collection fetches). Profile the cost drivers first; Supabase at the wrong access pattern gets expensive too.

The question nobody asked: is “we want relational queries” a genuine bottleneck, or two devs who'd simply rather work in Postgres? A legitimate preference — but name it as taste, not necessity. Add a reversible companion now, fix the cost driver now, and defer the irreversible Auth+Firestore migration indefinitely.

The Strategist Do it now — cost compounds and the data model ossifies — but scoped as dual-write over weeks. sonnet›

Six to twelve months out, “migrate now” looks very different depending on whether you count the cost you pay versus the cost you accrue. The cost of staying isn't flat, it's compounding: every new feature built on Firestore's document model becomes migration debt with interest. At 5,000 users you're migrating a database; at 25,000 with three more product surfaces, you're migrating a database and re-architecting the data-access layer under load, with customers noticing downtime. Headcount grows slower than migration surface area.

The upside nobody's pricing in: RLS + Postgres is an ops-load reduction for a team with no dedicated ops — standard SQL, standard tooling, a hireable talent pool — versus Firebase's rules DSL and bespoke tribal knowledge.

The real second-order risk is doing it now, badly, with no rollback plan. So the timing question isn't “now vs. later” on the calendar — it's “before or after the data model ossifies.” Do it now, but as dual-write-and-verify over weeks, never a cutover weekend.

The Executor Later — but “later” needs a trigger number, not a vibe. Do the cheap prep now. sonnet›

Two devs, no ops team, live production database — that's the whole decision. Migrate now: no. It requires dual-write/export scripts for every collection, a NoSQL-to-relational schema redesign, rewriting every Cloud Function as an Edge Function, re-implementing every Auth flow, writing and testing RLS policies that don't leak, and a cutover+rollback plan. That's 6–10 weeks of near-total roadmap freeze, minimum — and when RLS breaks at 2am, one person fixes it alone.

Stay forever: also no — the pain compounds. The only executable path is “later,” with a trigger: 20k users, $X/month spend, or a third engineer. Until then: stop adding Firestore collections that'll need joins, start a throwaway Supabase side-project so the team learns Postgres/RLS/Edge Functions, and track actual monthly Firebase cost.

Monday morning: write down the trigger number today; open a free Supabase project for an internal tool this quarter. Anything more ambitious with this headcount is roadmap suicide.

The Domain SpecialistReliability & Security lens Later — auth-hash migration is the landmine; runbook, hash-export test, rehearsed rollback first. sonnet›

Migrating live production auth and a live production database at the same time, with two generalists and zero ops bench, is the single highest-risk combination on this list — re-sequence before anything else.

Auth is the landmine. Firebase Auth password hashes use a scrypt variant with a secret hash key that isn't exported by default (request via the Admin SDK), in a modified-scrypt format Supabase's importer has historically had compatibility bugs with. Get it slightly wrong and users silently can't log in — failed hash verification doesn't throw, it falls through to “wrong password.” Test the full hash path against a prod copy first; never big-bang auth — run dual-auth (verify Firebase first, fall back to the migrated record, re-hash on login) for a full cycle.

Data: Firestore's denormalized documents don't map 1:1 to tables — design the schema, don't script a dump. RLS is not Firestore rules with new syntax: a missing policy fails safe (no access), but forgetting to enable RLS on a table fails catastrophically open to anyone with the anon key. Dual-write + reconciliation + feature-flagged reads + a rehearsed rollback.
